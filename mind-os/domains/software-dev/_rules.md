## 2. 渐进式对话协议

复杂功能不要一次性要求完整方案。遵循四步渐进：

```yaml
第1轮_问题空间: "先讨论不写代码。输入/输出/约束是什么？"
第2轮_方案对比: "2-3 个候选方案的 trade-off？"
第3轮_细节设计: "选定方向，展开数据结构和接口"
第4轮_可交付物: "写成代码骨架 / 实施指令"

反模式: 第1轮就要求"帮我把这个功能写出来"
```

---

## 3. 三层对话策略

| 层次 | 目的 | 策略 | 产出 |
|------|------|------|------|
| 设计讨论 | 架构决策、理论推导 | 长对话，允许探索反悔 | ADR文档、数据结构定义 |
| 规格编写 | 设计→可执行指令 | 中等对话，精确转译 | 四件套指令文档 |
| 实现+审计 | 代码落地+质量保证 | 短对话，审计另开新会话 | 代码 + 审计报告 |

**关键：审计必须在新会话中做。** 同一会话里 AI "记得自己做了什么"，倾向于确认而非质疑。新会话只能看实际代码，消除确认偏差。

## 4. 规格转译四件套

设计文档面向人类理解，AI 实现需要无歧义指令。转译必须包含：

```yaml
1_文件级动作清单:
  每个文件标明 DELETE / REPLACE / CREATE / MODIFY
  禁用模糊词："优化""重构""改进"

2_可验证检查点:
  每个检查点 = 一条可机器执行的命令
  例: "grep -r 'dp01' src/ 应返回 0 行"
  不是: "确保 X 正确"

3_禁令清单:
  明确"不允许做什么"
  例: "禁止创建包含 legacy/compat/fallback 的文件"
  例: "禁止 Object 代替 Map"

4_验证命令序列:
  Phase 完成后自动运行的检查脚本
  例: "npm run typecheck && npm run test && grep检查点"
```

## 5. AI 七种偏离模式（免疫清单）

| 偏差 | 表现 | 对策 |
|------|------|------|
| 安全优先降级 | 保留旧代码加适配层 | 先 DELETE 再 CREATE |
| 上下文溢出遗忘 | 后面忘了前面约束 | 每 Phase ≤ 15 文件变更 |
| 乐观完成申报 | 说"完成了"但有遗漏 | grep 检查点替代口头确认 |
| 类型降级 | 用 Object 代替 Map | 检查点明确要求类型 |
| 加兼容层 | 多余的 adapter/bridge | 禁令清单 |
| 字段缺失 | 少实现了可选字段 | 列出完整 interface |
| 双路径保留 | 新旧路径并存 | grep 旧特征词 = 0 行 |

## 7. 软件专属偏差检查

在 theory/think/rules.md 的通用偏差检查之外，软件开发场景额外触发：

```yaml
premature_optimization:
  trigger: "先优化性能"出现在功能未验证时
  protocol: 归零测试——"功能正确了吗？用户要了吗？"

sunk_cost_code:
  trigger: "已经写了很多"作为继续的理由
  protocol: 归零测试——"如果从零开始，还会选同样的方案吗？"

silver_bullet:
  trigger: "用了新技术/框架就能解决"
  protocol: 方法C——这个技术的逆向风险是什么？

complexity_bias:
  trigger: 方案层级 > 3 层 OR 抽象类 > 实现类
  protocol: 否定法——"能不能用更简单的方案？"

resume_driven:
  trigger: 技术选型理由中缺少用户价值
  protocol: 用户视角追问——"用户能感知到这个改进吗？"
```

## 8. 软件开发十二步流程

```yaml
# 设计阶段
step_1: 找本源——"这个问题在计算机科学/数学里叫什么？"
step_2: 提取原子操作——"最小的不可再分操作有哪些？"
step_2b: 同构检验——"这些操作中有本质相同的吗？"
step_3: 推演真实场景——"用户会这样操作：..."
step_4: 覆盖度审计——"哪些场景不能用原子操作覆盖？"
step_4b: 场景-原语螺旋——循环 step3→4 直到新场景不需要新原语
step_5: 识别空白——"诚实标记不确定的地方"
step_6: 实施计划——"分几期？先做什么？"

# 实现阶段
step_7: 规格转译——ADR → 四件套（动作+检查点+禁令+验证）
step_8: 实现调度——按风险拆 Phase，先删后建，每步可验证
step_9: 审计闭环——新会话审计，grep 举证，识别偏离
step_10: 知识沉淀——偏离模式→禁令模板，设计反馈→ADR更新

# 收敛判据
convergence: 新增 10 个极端场景，不需要新原语 → 原语集完备
```

## 9. 产业验证原则

复杂设计要找行业先例做交叉验证。行业验证的价值不是抄方案，而是**确认问题分解的方向正确**。

```yaml
protocol:
  1. 识别问题域中的标杆产品
  2. 分析其核心架构决策
  3. 对比自己的方案——一致=信心增强，冲突=需要解释原因
  4. "如果规模 ×10，还成立吗？"压力测试
```
