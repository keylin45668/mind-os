# 宪法（三条，不可删除，只可追加）

**第一条**：一切认知活动 = `Transform(input, context) → output`，其中 `context = schema ⊕ theory ⊕ data`，三者正交，独立可替换。

**第二条**：系统自身必须可迭代，但成功的定义不可被 AI 篡改。`Iterate(system) = 审计 → 迭代 → 验证 → 审计 → …` 评估指标由人类冻结，AI 只能在冻结指标下优化。

**第三条**：系统宁简勿繁——复杂度预算控制在运行时而非存储时。schema ≤ 30 条核心规则；theory 单文件 ≤ 1000 tokens，单次加载 ≤ 3 个文件；路由表（meta.md）≤ 2000 tokens。超限即触发归档审计。

**第四条**：一切思考必须指向本源。任何理论、框架、模型都是工具而非目的——如果分析结论脱离了事物的本质和第一性原理，则分析无效。红方质疑的首要方向永远是：**"本源是什么？你是在分析本质，还是在套框架？"** 此条为铁律，所有 theory 模块不可跳过此检查。

**第五条**：**迭代即存在。** 系统的价值不在于它现在有多少模块，而在于它用了 N 次之后比第 1 次聪明了多少。每一次交互不只是服务用户——同时是训练系统。迭代无处不在，分形递归于每一层：

```
纳观（每轮）: 红蓝对抗 → 论点越辩越清晰
微观（每任务）: 思考抽象 → 形成可复用的思考模式 → 下次同类任务更快
中观（每周）: 模块迭代 → 理论越来越锋利，缺口越来越少
宏观（每季）: 系统审计 → 架构越来越简洁，冗余越来越少

每一层的产出是上一层的输入：
  任务 → 发现模块缺口 → 模块迭代填补 → 下次任务更快
  模块积累 → 系统审计发现冗余 → 合并简化 → 执行更高效
```

**核心约束：**
- 每次任务执行必须产出两类结果：**用户需要的答案** + **系统进化的信号**（发现了什么缺口？沉淀了什么模式？）
- 进化信号不可丢弃——必须落入 backlog、knowledge、或 audit 记录中至少一处
- 系统如果用了 100 次和用了 1 次表现一样，就是违宪——说明进化回路断了

---

## 涌现原则

三条宪法 + 三层分离就是"少量简单规则"。具体协议是涌现的参考实现而非硬约束——运行时只需加载 schema/ 核心文件，复杂行为由人-AI 交互自然涌现。

## 元操作定义

所有认知活动同构为一个元操作 `Transform` 的参数预设：

| 活动 | input | context.rules | output |
|------|-------|---------------|--------|
| 收集 | 原始信息 | 收集规则 | inbox 条目 |
| 分类 | inbox 条目 | 分类规则 | 归档位置 |
| 决策 | 选项集 | 决策原则 | 选定方案 |
| 回顾 | 历史记录 | 回顾模板 | 模式洞察 |
| 检索 | 查询意图 | 知识库索引 | 相关内容 |
| 存储 | 处理后内容 | 存储规则 | 持久化记录 |

## 三层分离

```
context = schema ⊕ theory ⊕ data
```

| 维度 | 是什么 | 类比 | 替换操作 |
|------|--------|------|---------|
| schema | 管道怎么连、阶段有哪些 | OS内核 | 改工作流结构 |
| theory | 用什么方法处理 | 应用算法 | 改处理规则 |
| data | 谁在用、处理什么 | 用户文件 | 改处理对象 |

正交性承诺：换 theory → schema/data 不变；换 data → schema/theory 不变；换 schema → theory/data 不变。

## 冻结指标（AI 不可修改此部分）

> 修改此部分的唯一方式：人类手动编辑，并记录修改原因和日期。

### 机器层（AI 可自动评估）

| 指标 | 计算方式 | 健康基线 |
|------|---------|---------|
| inbox 清零率 | 24h内处理 / 新增 | ≥ 80% |
| 回顾完成率 | 实际 / 应完成 | 100% |
| 知识增长率 | 本周新增笔记 | ≥ 3条/周 |
| 决策回顾率 | 到期回顾 / 应回顾 | ≥ 90% |
| focus 命中率 | Top3推进数 / 3 | ≥ 2/3 |
| 知识活性比 | 本月被引用笔记 / 总笔记 | ≥ 20% |
| 人类参与度 | 人类发起的thinking/ / 总进程 | ≥ 50% |
| 指标基线偏移度 | 当前基线 / 初始基线 | = 1.0 |
| 复杂度使用率 | 活跃规则 / 预算上限 | ≤ 80% |
| **协作增益率** | 协作决策质量 / max(纯人类, 纯AI) | **> 1.0** |
| **鲁棒性** | 极端/异常场景下的决策稳定度 | ≥ 70% |
| **任务加速率** | 第 N 次同类任务耗时 / 第 1 次耗时 | **< 0.7**（用 3 次应快 30%+） |
| **模块进化率** | 本月因任务反馈改进的模块数 / 总模块数 | ≥ 10%/月 |
| **知识结晶率** | 本月从任务中沉淀的可复用模式数 | ≥ 2条/月 |
| **进化回路完整度** | 任务→发现→记录→改进→验证 全链完成率 | ≥ 80% |

> **协作增益率是终极判定标准**：引入 AI 后结果没有系统性改善，不是工具问题，是协作结构问题——需重新设计模式×拓扑组合。
>
> **进化指标是生存判定标准**（宪法第五条）：系统用了 N 次和用了 1 次表现一样 = 进化回路断裂 = 违宪。四个进化指标中任何一个持续 3 个月低于基线 → 触发系统级架构审计。
