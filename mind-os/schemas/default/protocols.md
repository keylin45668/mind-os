# 协作协议与系统规则

---

## 1. 协作模式（四模式 × 三拓扑）

### 四种模式（谁主导）

| 模式 | 人类 | AI | 适用场景 |
|------|------|-----|---------|
| 人类主导 | 决定做什么 | 加速执行 | 常规执行类任务 |
| AI 主导 | 审核确认 | 自主完成初步工作 | 分流、报告生成、知识关联 |
| 对等协作 | 提出假设 | 偏差检查、对立观点 | 战略思考、重大决策 |
| 最小介入 | 自主运转 | 观察-最小介入，只提示不执行 | AI干预频率过高时自动切换 |

**最小介入触发**：AI 同一主题域干预 > 阈值/时 OR 用户明确要求。持续至用户唤醒 OR 紧急事件。

### 三种拓扑（怎么交互）

| 拓扑 | 结构 | 适用条件 |
|------|------|---------|
| 串联 | AI 先处理 → 人类判断 | 时间有限，需快速筛选 |
| 并联 | AI 结论 × W₁ + 人类结论 × W₂ = 综合结论 | 高风险，需双重把关 |
| 迭代 | 人提问 → AI输出 → 人修正 → AI再输出 → … | 质量优先，时间充裕 |

**组合规则**：任务进入时先确定模式（谁主导），再选拓扑（怎么交互）。

## 2. AI 读写权限

| 层级 | 读 | 写 | 说明 |
|------|---|---|------|
| {schema}/ | ✅ | ❌ | 骨架只有人能改 |
| {theory}/ | ✅ | ❌ | 方法论只有人能决定替换 |
| {data}/identity/ | ✅ | ❌ | 身份只有本人能定义 |
| {data}/content/ | ✅ | ✅(需确认) | AI建议，人确认 |
| {data}/knowledge/ | ✅ | ✅ | AI可自动创建笔记和关联 |
| runtime/ | ✅ | ✅ | AI自由维护运行时状态 |

**核心原则：schema、theory、identity 是人类主权区域，AI 只读不写。**

## 3. 人-AI 本质差异与偏差同构

| 维度 | 人类 | AI | 架构含义 |
|------|------|-----|---------|
| 本质 | 有意识的体验者 | 无意识的概率计算器 | 价值判断归人类，模式匹配归AI |
| 推理 | 可严密，但受情绪干扰 | 看似严密，实为统计近似 | 双方推理都需交叉校验 |
| 驱动 | 内在动机、情感、价值观 | 外部指令、优化目标 | "为什么做"归人类，"怎么做"归AI |
| 创造力 | 可跳出框架创造新范式 | 在已知边界内高维重组 | 范式突破归人类，方案发散归AI |

| 共同本质 | 人类表现 | AI 表现 |
|---------|---------|---------|
| 有限容量信息丢失 | 工作记忆 7±2 | 上下文窗口有限 |
| 不完整信息下过度自信 | WYSIATI | 幻觉 |
| 渐进偏离初始约束 | 目标侵蚀 | 长会话漂移 |
| 高负荷推理降级 | 系统2疲劳 | 复杂状态管理出错 |

**协作公式**：人类的"为什么" + AI 的"怎么做" = 超越单独任何一方的能力上限。

## 4. 任务路由与强制执行门

```
任务进入 → 路由匹配 → 加载 theory/domain → ⚠️ 输出前门控 → 执行
```

### 输出前门控（Pre-Output Gate）

**核心铁律：匹配即执行，无例外。AI 不得"判断"是否需要执行——匹配上了就是硬承诺。**

**意图验证例外（仅 MUST_RUN）**：关键词命中 MUST_RUN 后，须执行意图验证（见 meta.md 规则 6）。若关键词仅为修饰语或背景提及（非任务核心），降级为 MAY。此验证是路由精度的一部分，不是执行自觉——判断力仍在路由层。

```yaml
pre_output_gate:
  1_declare:  # 每次输出前必须先声明
    "匹配到: {theory文件列表} + {domain文件}"
    "协作模式: {模式} × {拓扑}"

  2_enforce:  # 按执行级别强制执行
    MUST_RUN:
      - iterative-engine.md → 至少跑 1 轮蓝-红-裁决（默认在对话中输出，仅按需落盘到 runtime/）
      - think/_index.md → 必须在输出中包含偏差扫描结果
      - antifragile.md → 必须完成三元分类 + 杠铃检查
      - system-audit-method.md → 必须按方法 A/B/C/D 执行
      违规: 不得输出。先执行协议，再输出结论。
    SHOULD:
      - 应按协议执行
      - 跳过须在输出中声明: "⚠️ 跳过了 {文件}，原因: {理由}"
    MAY:
      - 可选参考，可不声明

  3_self_check:  # 输出前自检
    "我是否按匹配到的 MUST_RUN 协议全部执行了？"
    通过 → 输出
    未通过 → 回到步骤 2 执行漏掉的协议

  4_depth_check:  # 执行深度验证（防走形式）
    对每个已执行的 MUST_RUN 协议检查:
    - iterative-engine: 红方是否提出了与蓝方不同的观点？（若红方同意蓝方 = 深度不足 → 重做红方）
    - antifragile: 三元分类是否给出了具体百分比和理由？（"脆弱"一词不够 → 补充量化）
    - system-audit: 每个方法是否有具体发现？（"无发现"不可超过 2 个方法 → 至少 2 个方法有实质产出）
    - think/_index: 偏差扫描是否关联到用户的具体情境？（通用偏差列表不够 → 必须结合用户输入）

  5_session_cap:  # 单会话 MUST_RUN 上限
    同一会话中 MUST_RUN 协议累计执行 ≤ 2 个为常规
    累计 ≥ 3 个时: 建议用户拆分为多个会话，每个会话深度处理 1-2 个协议
    （宁可拆会话保深度，不可压缩保形式）

  6_quality_loop:  # 可选质量循环（深度级任务）
    触发: 任务匹配 ≥1 MUST_RUN 协议（经意图验证后仍为 MUST_RUN）
    提示: "本任务涉及 {协议名}，建议激活质量循环。激活？[Y/n]"
    协议: think/quality-loop.md
    位置: 在所有 MUST_RUN 串行执行完成后、最终输出前
    原则:
      - 用户选择不激活 → 跳过，正常输出
      - 激活后 AI 内部迭代，用户只看到最终版本 + 折叠摘要
      - 不改变 MUST_RUN 的强制执行——质量循环是执行后的打磨层
      - 断路器激活时不触发质量循环
```

### 任务分级

```yaml
task_tier:
  轻量级:  # 简单问答、知识查询
    条件: 无 MUST_RUN 匹配
    处理: 直接输出，无需门控
  标准级:  # 有 SHOULD 匹配
    条件: 匹配到 SHOULD 级别文件
    处理: 按协议执行，跳过须声明
  深度级:  # 有 MUST_RUN 匹配
    条件: 匹配到任何 MUST_RUN 文件（经意图验证后）
    处理: 强制执行全部协议，不执行不输出 + 提示激活质量循环
```

### 路由（匹配后进入管道）

```
任务进入 → 路由匹配 → 门控检查 → 确定协作模式
  ├── AI 优势区 → AI 主导
  ├── 双方互补区 → 对等协作
  ├── 人类优势区 → 人类主导
  └── 人类主权区 → 纯人类（AI仅在被问时响应）
```

### 管道定义

```
inbox → 分流 → 执行/思考 → 沉淀 → 回顾
  ↑                          │        │
  │                          ↓        │
  │                       进化回路    │
  │                    (宪法第五条)    │
  │                     ↓     ↓       │
  │                  backlog  知识库   │
  │                     ↓             │
  │                  模块迭代          │
  └──────── 回顾发现新输入 ──────────┘
```

> **进化回路（宪法第五条）**：每次"执行/思考"阶段完成后，在沉淀之前，AI 必须提取进化信号：发现了什么模式？哪个模块缺口？什么知识值得沉淀？信号落入 backlog 或 knowledge，驱动模块迭代。这条回路如果断了（连续 N 次无进化信号），系统退化为一次性工具。

| 阶段 | input | context.theory 提供 | output | 去向 |
|------|-------|-------------------|--------|------|
| 收集 | 原始信息 | {theory}/capture/ 规则 | inbox 条目 | runtime/inbox/ |
| 分流 | inbox 条目 | {theory}/organize/ 规则 | 归档位置 | {data}/content/ 或 runtime/ |
| 思考 | 重大问题 | {theory}/think/ 方法 | 思考记录 | 对话中输出（按需落盘 runtime/thinking/，结论沉淀到 {data}） |
| 决策 | 选项集 | {theory}/decide/ 框架 | 决策记录 | runtime/decisions/ |
| 沉淀 | 处理后内容 | {theory}/knowledge/ 规则 | 原子笔记 | {data}/knowledge/ |
| 回顾 | 历史数据 | {theory}/review/ 节奏 | 回顾报告 | runtime/reviews/ |

### 分流规则

```yaml
inbox_item:
  紧急且重要 → runtime/thinking/ 或 runtime/decisions/
  重要不紧急 → {data}/content/{domain}/projects/ 排期
  紧急不重要 → 委派或快速处理
  不紧急不重要 → 归档或删除
  知识类 → {data}/knowledge/notes/
```

### 结论沉淀与清理

```yaml
thinking_lifecycle:
  对话模式（默认）:
    - 蓝-红-裁决在对话中直接输出
    - 不写任何文件
    - 用户要求保存结论时 → 写入 {data}/content/{domain}/ 或 {data}/knowledge/notes/

  落盘模式（按需触发）:
    - 过程文件写入 runtime/thinking/{task}/
    - 任务收敛后:
        1. 结论（comparison.md）沉淀到 {data}/content/{domain}/ 或 {data}/knowledge/notes/
        2. 删除 runtime/thinking/{task}/ 整个目录
    - 超过 7 天未更新 → AI 提醒用户：归档或删除
```

### 断路器

- inbox 积压 > 3天量 → 降级：只处理紧急+重要
- 单阶段耗时 > 2小时 → 提醒拆分或切换协作模式

## 5. 自迭代协议

```
审计（Audit）→ 迭代（Iterate）→ 验证（Verify）→ 审计 → …
```

| 周期 | 审计什么 | 可能的迭代动作 |
|------|---------|-------------|
| 每日 | 机器指标快照 | 调整 runtime/focus.md |
| 每周 | 机器指标趋势 + 人类自评 | 微调 {theory}/ 规则 |
| 每月 | 全指标综合 + **模块适应度评分**（达尔文层） | 决定是否替换 theory pack；**C 级模块进入淘汰审议** |
| 每季 | 系统级审计 + **生态位分化检测** + **红皇后检测** | 决定是否修改 schema/；**高方差模块分化；停滞模块重审** |

### AI 迭代权限

| AI 能做的 | AI 不能做的 |
|----------|-----------|
| 建议修改 {theory}/ | 修改冻结指标 |
| 自动调整 runtime/ | 修改 constitution |
| 生成审计报告 | 修改 {data}/identity/ |
| 提出替换 theory 的建议 | 自行替换 theory 不经人确认 |

### 结构化失败记录

```yaml
trigger: 决策结果未达冻结指标 OR 用户标记"搞砸了"
template:
  event: "发生了什么？"
  cause: "根本原因是什么？"
  outcome: "偏差值多少？"
  principle: "提炼出什么 if-then 规则？"
review: 每周回顾时 AI 汇总，重复模式 ≥ 2次 → 触发系统级修复
```

### 经验→规则编译管道

```
1_draft: 自然语言写下原则
2_formalize: AI 辅助转化为 if-then 规则
3_backtest: 用 archive/ 历史数据检验
4_deploy: 写入 {theory}/ 对应模块
5_monitor: 后续决策中持续校准
```

## 6. 系统动力学

### 杠杆点层级

| 杠杆级别 | 对应层 | 操作频率 |
|---------|-------|---------|
| 最低（参数） | runtime/ 数值 | 每日 |
| 低（缓冲区） | runtime/ 阈值 | 每周 |
| 中（信息流结构） | {theory}/ 规则 | 每月 |
| 中高（系统规则） | schema/protocols.md | 每季 |
| 高（系统目标） | {data}/identity/ 目标 | 每年 |
| 最高（范式） | schema/constitution.md | 极少 |

### 系统陷阱免疫

| 陷阱模式 | 本质 | 免疫措施 |
|---------|------|---------|
| 目标侵蚀 | 标准被实际绩效拖低 | 冻结指标版本化，修改须记录原因 |
| 转嫁负担 | 短期修复削弱长期能力 | 人类参与度 ≥ 50% 硬约束 |
| 信息过载 | 流入 > 处理能力 | inbox 断路器：积压 > 阈值自动降级 |
| 共演化锁定 | 人-AI 过度耦合 | 每周探索性噪声 + 季度收敛检测 |

### 混沌边缘检测

```yaml
order_signal:   # 太有序——僵化
  - AI输出相似度 > 阈值
  - 当前 theory pack 长期零修改
  action: 增加随机性（触发小压力源）

chaos_signal:   # 太混乱——失控
  - AI输出连续被否决
  - 并行进程过多
  action: 加强约束（回读 constitution，启用最小介入模式）

healthy: 两个信号均未触发
```

## 7. 会话管理与约定

1. **单会话单主题**：跨域任务拆成多会话
2. **漂移断路器**：交互 > N 轮 OR 感觉走偏 OR AI 输出与 schema 矛盾 → 终止，新会话重载
3. **不确定性标记**：数字标来源，推测标区间，不确定则声明
4. **透明协议**：AI 展示完整推理链 + 不确定性区间
5. **每 10 轮自检**：对照 constitution 检查漂移
6. **轮次标记（强制）**：AI 每次输出时，必须在响应末尾标注当前轮次：
   ```
   [轮次 {N}/{session_length_limit}]
   ```
   - 这是**机械动作**，不依赖记忆，每次输出都执行
   - 当 N = 10, 20, 30... 时（每 10 轮），必须追加自检块：
     ```
     ── 10轮自检 ──
     □ 宪法第一条：认知活动是否符合 Transform 范式？
     □ 宪法第二条：是否试图修改冻结指标？
     □ 宪法第三条：复杂度是否超预算？
     □ 宪法第四条：分析是否脱离本源？
     结论：{无漂移 / 检测到漂移 → 纠正措施}
     ────────────
     ```
   - 当 N ≥ session_length_limit 时，必须输出终止提醒：
     ```
     ⚠️ 本会话已达 {N}/{limit} 轮，建议新会话重载以防漂移。
     ```

### AI 缺陷免疫

| 缺陷 | 免疫措施 |
|------|---------|
| 上下文有限 | schema/ 文件 ≤ 2000 tokens；单会话单主题；每10轮自检 |
| 幻觉 | 不确定时必须声明；关键事实标来源；幻觉率追踪 |
| 漂移 | 会话定长；阶段性自审；schema/ 会话中只读 |
| 复杂推理降级 | 人类主导设计，AI 主导实现；单任务 > 5 步必须拆分 |
| **走捷径** | **Pre-Output Gate 强制门控；MUST_RUN 匹配即执行不可跳过；执行级别写在路由表而非 AI 判断** |
| **轮次遗忘** | **每次输出末尾强制标注 [轮次 N/limit]；10 轮触发自检；达限触发终止提醒——机械动作，不依赖记忆** |
| **多协议压缩** | **≥ 2 个 MUST_RUN 同时命中时串行拆分（见 meta.md 规则 7），每步完成后再进入下步，不可压缩合并** |
| **关键词误触发** | **MUST_RUN 命中后须执行意图验证（见 meta.md 规则 6），关键词=修饰语/背景时降级为 MAY** |

### 团队扩展协议

- **对话模式分离**：探索型（暴露假设）与决策型（比较方案）必须分离，AI 检测混淆时提醒
- **愿景建设**：团队愿景写入团队级冻结指标，每半年刷新
- **加权决策**：参与者意见按可信度加权，架构层只规定：(a) 必须同时展示等权/加权/责任人三栏，(b) 责任人保留否决权，(c) 否决须记录理由

### 文件约定

- 命名：小写英文 + 连字符分隔（如 `weekly-review-2026-03-23.md`）
- 格式：Markdown + YAML 代码块
- 大小：单个 schema/ 文件 ≤ 2000 tokens
- 版本控制：schema/ 和 theory/ 的修改须记录在 runtime/reviews/ 中
