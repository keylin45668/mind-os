# 系统审计方法论

> 来源：《AI 协作思维框架 v2.0》
> 用途：系统自迭代（schema/iteration.md）中"审计"原语的具体方法

---

## 1. 两条审计路径

### 路径 A：本源先行（Top-Down）

```
本源挖掘 → 方案生长 → 场景验证 → 闭环
"这到底是什么？" → "怎么做？" → "做对了吗？" → "落地"
```

适用：问题域有明确的数学/理论对应物。先找到本源，再从本源生长方案。

### 路径 B：方案先行，本源后追（Bottom-Up Peeling）

```
直觉方案 → 审计发现缺口 → 追问更深本源 → 方案重构 → 再审计 → …
```

适用：问题域复杂、本源不明显、需先做出可审计的方案再"剥洋葱"。

**路径 B 不是偷懒——方案本身是找到本源的工具。两条路径最终收敛到同一终态。**

判断用哪条：
- 5 分钟内能说出数学本质 → 路径 A
- "大概知道想要什么但说不清本质" → 路径 B
- 不确定 → 默认路径 B（更安全）

## 2. 四种向下挖掘方法

### 方法 A：数学形式化

问"这个东西在数学里叫什么？"

```yaml
trigger: 新概念/新规则引入时
protocol:
  1. 能否写成严格定义（集合、映射、公理）？
  2. 能 → 形式化后检验是否与已有定义冲突
  3. 不能 → 标记为"直觉层"，记录待形式化
用于系统审计: 检验新 theory/ 规则是否可形式化表达
```

### 方法 B：原子操作分解

问"这件事最小的不可再分操作是什么？"

```yaml
trigger: 新模块/新流程设计时
protocol:
  1. 持续拆分直到不可再分
  2. 原子操作数 ≤ 7（认知负荷上限）
  3. 最小性检验：去掉任意一个，哪些场景做不了？
  4. 完备性检验：增加一个，是不是已有组合可覆盖？
用于系统审计: 检验 schema/ 规则是否已拆到原子级
```

### 方法 C：正反对称检验

找到正向操作后，问"逆向操作是什么？"

```yaml
trigger: 任何单向流程定义时
protocol:
  1. 正向操作 → 逆向操作是什么？
  2. 逆向操作存在且完整 → ✅
  3. 逆向操作不存在或不完整 → 设计有遗漏
用于系统审计:
  - 换 theory → schema/data 不变？（正向）
  - 换回原 theory → 系统恢复原状？（逆向）
  - 同理测试换 data、换 schema
```

### 方法 D：同构检验

问"这几个看起来不同的东西，在本质上是不是同一个？"

```yaml
trigger: 系统中出现≥2个相似概念/规则时
protocol:
  1. 列出每个概念的核心属性
  2. 属性匹配度 > 80% → 同构，应合并
  3. 合并后检验是否丢失了区分度
  4. 方法 D 是方法 B 的逆操作——B 拆，D 合
用于系统审计:
  - theory/ 中是否有重复模块？
  - 新增内容与已有内容是否同构？
  - 同构 → 合并，不新增模块数
```

## 3. 焦距交替

设计和审计中每 2-3 轮主动切换视野：

```yaml
望远镜（全局视野）: 看结构一致性、原则对齐、层级关系
显微镜（局部深入）: 看具体场景、边界条件、交互细节

切换信号:
  - 细节讨论中感到"有更大的东西没对" → 切望远镜
  - 宏观讨论中感到"具体怎么工作？" → 切显微镜
  - 连续两轮方案被否决 → 焦距可能不对
```

**望远镜发现结构问题，显微镜发现边界条件。只用一种焦距会遗漏另一类问题。**

## 4. 深度追问模板

### 向下追问（找更深的本源）

- "这是最底层的原语了吗？还能再拆/再合并吗？"
- "这几个看似不同的概念，底层是同一个吗？"
- "这个参数是不是被硬编码了？应该是可配置的吗？"
- "去掉这个规则，哪些场景就不工作了？"

### 向上追问（找更广的组合能力）

- "用户能用这些规则组合出任意场景吗？"
- "如果不能，缺哪类规则？"
- "这个设计能复用吗？怎么跨领域共享？"

### 横向追问（找遗漏的维度）

- "这个设计有逃逸口吗？逃逸口违反宪法吗？"
- "如果改了 A，B 会自动跟着变吗？如果不会，耦合没解干净"
- "规模 ×10，还能用吗？"

### 用户视角追问

- "零基础用户怎么上手这套系统？"
- "高级用户嫌不够用时怎么扩展？"
- "用错了怎么回退？"

## 5. 约束晚注入协议

最重要的约束往往在讨论中途才被意识到——不是遗忘，是早期抽象层次不够深。

```yaml
protocol:
  1. 识别约束层级:
     普通约束: 只影响局部设计
     宪法级约束: 影响已完成的全部设计
  2. 宪法级约束处理:
     a. 用此约束审计已有文档每一处，标记违规点
     b. 逐处修正——不是打补丁，是用约束的语言重述
     c. 将约束提升到 schema/constitution.md 级别
     d. 确保后续内容天然对齐
  3. 接受沉没成本:
     被推翻的内容的价值 = 通过写它才发现了这条约束
```

## 6. 季度系统审计流程

```yaml
quarterly_audit:
  1. 方法 C 正交性测试:
     - 模拟换 theory → schema/data 零改动？
     - 模拟换 data → schema/theory 零改动？
     - 模拟换 schema → theory/data 零改动？
  2. 方法 D 冗余扫描:
     - theory/ 模块间有同构内容？→ 合并
     - schema/ 规则间有重叠？→ 精简
  3. 方法 B 原子性检查:
     - 每条 schema 规则是否已是最小单元？
     - 每个 theory 模块是否可独立替换？
  4. 焦距交替:
     - 望远镜：整体架构是否还清晰？
     - 显微镜：抽查 3 个具体决策场景走完整流程
  5. 深度追问:
     - "这套系统还能再简化吗？"（否定法）
     - "哪些规则从未被触发？"（死代码清理）
     - "规模 ×10 还能用吗？"（压力测试）
```
