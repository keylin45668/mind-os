# Sentinel Protocol — Mind OS 智能意图检测层设计

> 日期：2026-03-26
> 状态：设计已确认，待实施

## 1. 问题

当前 mind-os 的所有触发路径都依赖用户主动操作（关键词、斜杠命令、显式启动）。典型遗漏场景：

- 做重要决策时直接开始讨论，忘了启用深度分析
- 进入项目仓库工作，忘了加载 connector
- 会话快结束时，忘了触发 `/评分`

## 2. 设计目标

- 在现有架构上增加"智能意图检测层"，不改动核心架构
- 工具无关 — 纯 markdown 协议，Claude Code / Cursor / Gemini CLI / Codex / Trae 均可执行
- 半自动：检测到后带上下文建议，用户确认后才加载
- 可关闭、不烦人

## 3. 方案：sentinel.md 检测协议

### 3.1 定位

```
sentinel.md = 检测 + 建议（"你可能需要这个"）
meta.md     = 路由 + 执行（"加载并运行这个"）
```

sentinel 是被动防线，不执行任何理论模块，只负责检测场景并向用户建议。用户确认后走 meta.md 已有路由。

**与 meta.md 的互斥原则：** sentinel 只在 mind-os 未启动（即 meta.md 未加载）时做场景检测。如果用户已通过 BOOT 或手动命令启动了 mind-os，meta.md 的关键词路由已经在工作，sentinel 的场景信号检测自动停用，仅保留上下文信号和时间信号。这从根本上避免双重触发。

### 3.2 文件位置

`mind_os/mind-os-core/sentinel.md`

### 3.3 加载时机

**项目仓库场景（通过 .mind-os.md）：**

```
1. READ connector
2. READ constitution.md（前5行验证）
3. READ sentinel.md（加载检测规则到会话上下文）  ← 新增
4. 输出连接提示
```

**非项目场景（通过 BOOT.md）：**

```
Phase 4 加载列表增加 sentinel.md
```

加载后常驻会话上下文，不需每轮重新读取。

**上下文持久性说明：** "常驻"是最佳情况假设，实际取决于各工具的上下文窗口管理。sentinel.md 自身包含一条自检指令：如果 AI 无法回忆检测规则，应重新 READ sentinel.md。由于文件极小（< 400 tokens），重新加载成本可忽略。

**BOOT 状态判定：** sentinel 需要判断"mind-os 是否已通过 BOOT 启动"。判定方式为行为启发式：如果本会话中曾显示过 BOOT 启动面板（Phase 5 输出），则视为 meta.md 路由已激活，sentinel 的场景信号和时间信号中与 meta.md 重叠的部分自动停用。这是一个二元状态（启动过 / 未启动过），不会因上下文压缩而丢失。

### 3.4 Token 预算

目标 < 400 tokens。对比 meta.md 约 2000 tokens，sentinel 只有其 1/5，开销可忽略。

**预算保护：** sentinel.md 不纳入 autoevolve 模块迭代范围。如需修改检测规则，通过人工编辑完成。这确保文件大小不会因自动迭代而膨胀。

## 4. 检测规则体系

### 4.1 场景信号（关键词 + 句式模式）

| 场景 | 关键词 | 句式模式 | 建议模块 | 理由模板 |
|------|--------|----------|----------|----------|
| 决策 | 决策/选择/取舍/权衡 | "A还是B" / "要不要X" / "该怎么选" / "值不值得" | /偏差 + /反脆弱 | 涉及重要决策，建议检查认知偏差和反脆弱性 |
| 分析 | 分析/想清楚/深度/本质/根因 | "为什么会X" / "到底是什么问题" | /分析 | 检测到深度分析需求 |
| 风险 | 风险/投资/融资/亏损/对冲 | "万一X怎么办" / "最坏情况" | /反脆弱 | 涉及风险评估 |
| 规划 | 排期/计划/路线图/里程碑 | "接下来怎么做" / "分几步" | /排期 | 涉及任务规划 |
| 回顾 | 复盘/回顾/总结/反思 | "上次为什么X" / "哪里做得不好" | /回顾 | 涉及经验回顾 |
| 创意 | 创意/脑暴/灵感/点子 | "有什么好办法" / "还能怎么做" | /创意 | 涉及创意发散 |
| 沟通 | 沟通/谈判/说服/汇报 | "怎么跟X说" / "怎么让X同意" | /沟通 | 涉及沟通策略 |

**匹配逻辑：**
- 关键词命中 OR 句式模式命中 → 触发建议
- 两者同时命中 → 措辞更确定
- 仅句式命中 → 措辞温和（"看起来可能涉及..."）

### 4.2 上下文信号（环境检测）

**BOOT 流程专用：**

| 信号 | 检测条件 | 动作 |
|------|----------|------|
| 进入项目仓库 | cwd 包含已注册项目路径 | 建议加载对应 connector |
| 首次提及已注册项目 | 对话中出现项目 connector 的 keywords | 建议加载 connector |
| mind-os 已启动但未加载项目 | BOOT 后状态，用户谈及具体项目 | 建议加载对应 connector |

> 数据源：BOOT Phase 4 扫描 `local/projects/*.md` 获得已注册项目列表。仅 BOOT 流程可用。

**桥接文件流程专用：**

| 信号 | 检测条件 | 动作 |
|------|----------|------|
| 当前仓库有 .mind-os.md | 检测 cwd 下存在 .mind-os.md | 建议启动 mind-os 连接 |

> 桥接文件流程不经过 BOOT，无已注册项目数据，仅能检测当前仓库是否有 `.mind-os.md`。

### 4.3 时间信号（会话生命周期）

| 信号 | 检测条件 | 动作 |
|------|----------|------|
| 会话结束 | 关键词："再见/结束/今天就到这/bye/下次再说"，且本会话未经过 BOOT 启动流程（即未显示过启动面板） | 建议执行 /评分 |
| 长会话无回顾 | 对话轮数 > 20 且未触发过回顾类模块 | 轻量提醒阶段性回顾 |

## 5. 建议输出格式

```
> 🧠 Mind OS 建议
> 检测到：{场景描述}（{触发原因}）
> 建议加载：{模块列表}
> 理由：{具体说明}
> → 回复"好"加载 / "不用"跳过
```

示例：

```
> 🧠 Mind OS 建议
> 检测到：决策场景（句式"要不要扩招三个人"+ 关键词"取舍"）
> 建议加载：/偏差 + /反脆弱
> 理由：涉及人事决策和资源投入，建议检查认知偏差并做反脆弱分析
> → 回复"好"加载 / "不用"跳过
```

用户确认后，直接走 meta.md 已有路由加载理论文件并执行。

## 6. 抑制机制

| 规则 | 说明 |
|------|------|
| 同场景单次 | 同一场景类型在一个会话中最多建议一次（/评分 除外） |
| 已激活不重复 | 已加载的模块（无论手动还是自动路由），sentinel 不再建议 |
| 拒绝后降级 | 用户说"不用"后，同类场景本会话静默。不跨会话记忆 |
| 不打断 | 建议只在用户输入后、AI 回复开头出现 |
| 评分例外 | /评分 提醒不受单次限制，但已执行过则不再提醒 |

## 7. 用户控制

- **关闭**："不用提醒了" / "别建议了" / "安静"  → 本会话全部静默
- **重新开启**："继续提醒" / "恢复建议" → 恢复检测

## 8. 对现有文件的改动

| 文件 | 改动 | 大小 |
|------|------|------|
| `mind-os-core/sentinel.md` | **新增** | ~400 tokens |
| `.mind-os.md`（各项目） | 启动流程步骤 3 增加：`READ {mind_os}/mind-os-core/sentinel.md` | +1 行 |
| `BOOT.md` Phase 4 | 加载列表增加 `sentinel.md`（与 meta.md 同级目录） | +1 行 |
| `meta.md` | 不改 | — |
| `constitution.md` | 不改 | — |
| connector 文件 | 不改 | — |

## 9. 设计原则

- **最小侵入**：1 个新文件 + 各入口加 1 行，不改核心架构
- **工具无关**：纯 markdown 协议，任何能读文件的 AI 工具都能执行
- **渐进增强**：未来可按需为特定工具加 hook 硬层，但不是必需的
- **克制**：宁可漏检也不误报，抑制机制保证不打扰
