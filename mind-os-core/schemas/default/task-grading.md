# 任务分级与准入门控

## 集成说明

- **在流水线中的位置**（见 `protocols.md §4`）：`路由匹配 → 任务分级 → Pre-Input Gate → 加载 theory → Pre-Output Gate → 执行`
- **BOOT.md**：Phase 2 完成路由后、执行前，插入 Pre-Input Gate 检查。分级结果决定加载上限和门禁数量。
- **constitution.md**：本文件遵守第三条（≤ 30 规则），总规则数 = 3 级定义 + 7 项门检 + 3 条加载上限 = 13 条。

---

## 1. 三级任务分级

| 级别 | 标识 | 条件 | 处理路径 |
|------|------|------|---------|
| 轻量思考 | 🟢 | 单模块、无高风险、无跨项目、无不可逆决策 | 直接走 theory routing，串联拓扑 |
| 标准思考 | 🟡 | 涉及决策/风险评估/多领域交叉/跨模块 | red-blue 对抗 + 人工确认门禁，并联拓扑 |
| 深度思考 | 🔴 | 涉及战略/不可逆决策/跨多项目/范式级变更 | 多轮迭代 + 分阶段交付 + 新窗口复审，迭代拓扑 |

### 分级判定规则

```yaml
grading_rules:
  default: 🟢  # 未命中任何升级条件时默认轻量
  upgrade_to_🟡:
    - 涉及 ≥ 2 个 theory 模块
    - 包含风险评估或权衡决策
    - 跨领域（如 decide + think 同时命中）
    - 用户显式要求"仔细想"/"深入分析"
  upgrade_to_🔴:
    - 涉及不可逆决策（资源分配、战略方向、关系处理）
    - 跨 ≥ 2 个项目或身份角色
    - 用户显式要求"战略级"/"全面审视"
    - 🟡 任务执行中发现复杂度超预期 → 动态升级
```

## 2. Pre-Input Gate（准入检查）

**触发时机**：routing 完成、任务分级确认后，执行前。

```yaml
pre_input_gate:
  checks:
    1_context_readable: "所需 theory 文件是否存在且可读？"
    2_grade_confirmed: "任务分级是否已确定？（🟢/🟡/🔴）"
    3_project_connector: "涉及的项目连接器（.mind-os.md）是否就绪？"
    4_budget_check: "theory 加载数是否在本级上限内？"
    5_session_scope: "任务是否符合单会话单主题？跨主题须拆分"
    6_data_permission: "所需 data 层文件的读写权限是否满足？"
    7_irreversibility_flag: "是否涉及不可逆操作？是 → 强制升级为 🟡+"
  policy:
    all_pass: 继续执行
    any_fail: 阻断，报告未通过项，等待修复或人工确认
```

## 3. 分级加载上限与门禁

| 级别 | theory 加载上限 | Pre-Input Gate 必检项 | Pre-Output Gate | 协作模式 |
|------|----------------|----------------------|-----------------|---------|
| 🟢 | ≤ 2 个 | #1 #2 #4 | 现有规则（无 MUST_RUN 直接输出） | 人类主导/AI 主导 |
| 🟡 | ≤ 3 个 | 全部 7 项 | SHOULD 级 + red-blue 对抗 | 对等协作（并联） |
| 🔴 | ≤ 3 个（分阶段多次） | 全部 7 项 + 每阶段重检 | MUST_RUN + 质量循环 + 新窗口复审 | 对等协作（迭代） |

## 4. 动态升降级

- **升级**：执行中发现未预见的风险/复杂度 → 暂停，升级分级，重新过 Pre-Input Gate
- **降级**：🟡 任务经 red-blue 确认风险可控 → 人确认后可降为 🟢 继续
- 所有升降级须在 runtime 状态中记录原因
