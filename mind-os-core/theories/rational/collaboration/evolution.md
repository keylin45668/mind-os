# 模块进化机制（达尔文层）

> 来源：《物种起源》选择压力 + 适应性景观 + 生态位分化
> 定位：与拉马克层（宪法第五条，任务级学习）互补——拉马克负责学习，达尔文负责淘汰
> 触发：每月回顾时自动执行，或 /进化 手动触发

---

## 1. 模块适应度评分

```yaml
fitness_score:
  计算周期: 每月
  公式: |
    fitness = (命中率 × 0.3) + (用户评分贡献 × 0.3) + (任务加速贡献 × 0.2) + (新鲜度 × 0.2)

  维度:
    命中率: 本月被路由命中次数 / 总任务数
    用户评分贡献: 使用该模块的会话 D2 均分 / 全局 D2 均分
    任务加速贡献: 使用该模块的同类任务是否越来越快
    新鲜度: 1 - (距上次修改天数 / 90)，下限 0

  等级:
    S (≥0.8): 核心模块，优先维护
    A (0.6-0.8): 健康模块
    B (0.4-0.6): 观察区——需关注是否在退化
    C (<0.4): 归档候选——连续 2 月 C 级触发淘汰审议
```

## 2. 淘汰归档（灭绝）

```yaml
extinction_protocol:
  触发: 模块连续 2 个月 fitness = C 级
  流程:
    1. AI 生成淘汰报告: 为什么低分？是没用到还是用了不好？
    2. AskUserQuestion:
       question: "{模块名} 连续 2 月低适应度，怎么处理？"
       options:
         - label: "归档"
           description: "移入 theories/{pack}/archive/，路由表移除"
         - label: "合并"
           description: "内容并入其他模块（方法 D 同构）"
         - label: "保留观察"
           description: "再给 1 个月，但降低路由优先级"
         - label: "重构"
           description: "问题不在内容而在结构，/模块迭代 重构"
    3. 人类决定后执行
  约束: MUST_RUN 级别的模块不可归档（偏差检查、迭代引擎等核心模块豁免）
```

## 3. 生态位分化（物种形成）

```yaml
speciation_trigger:
  检测: 同一模块在不同 domain 下的用户评分方差 > 1.5（5 分制）
  含义: 模块太通用，在某些场景好用但另一些不好用
  示例: antifragile.md 在投资场景 4.5 分，团队管理场景 2.8 分

  流程:
    1. AI 分析: 高分场景和低分场景的差异在哪？
    2. 建议: "antifragile.md 在 {domain1} 和 {domain2} 表现差异大，建议分化"
    3. AskUserQuestion:
       question: "{模块名} 是否需要针对不同场景特化？"
       options:
         - label: "分化"
           description: "拆分为 {模块-domain1}.md + {模块-domain2}.md"
         - label: "优化通用版"
           description: "不拆分，改进通用性"
         - label: "先不动"
           description: "数据不够，继续观察"
    4. 分化时复制原模块，各自添加场景特化规则
```

## 4. 红皇后效应（反停滞）

```yaml
red_queen:
  原理: "你必须不停奔跑，才能留在原地"——用户需求在变，模块不进化=退化
  检测: 模块 fitness 连续 3 月不变（既没涨也没跌）
  含义: 可能"够用了就不改了"，但用户需求可能已经变了
  操作:
    - 标记为"停滞"
    - 下次涉及该模块的任务时，AI 在审查中提问:
      "这个模块上次修改是 {N} 天前，内容还适用当前场景吗？"
    - 不强制修改，只确保人类意识到停滞
```
