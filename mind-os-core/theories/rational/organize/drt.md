## 摘要

- 任何工作 = Why(决策) × What(需求) × How(任务)，三维正交
- 上一层的 How 展开后就是下一层的 Why（分形递归）
- 维度之间通过"共振点"双向触发，非线性

# DRT 正交模型

> **本源公式：** `Work(item) = Why(决策) × What(需求) × How(任务)`

---

## 触发条件

```yaml
trigger:
  - 用户提到"工作"、"任务"、"需求"、"决策"等管理类话题
  - 用户在拆解工作、分配任务、定义需求时
  - 用户感到工作推进受阻、方向不清时
```

## 协议

### 1. 入口定位（必做）

对当前工作执行两步定位：

```
Q1: 什么尺度？
    战略(季度影响) / 功能(Sprint影响) / 实现(天级) / 代码(小时级)

Q2: 从哪个维度进来的？
    "我们要不要做X"       → 从 Why 进入
    "客户要一个Y功能"     → 从 What 进入
    "这个bug得修一下"     → 从 How 进入
```

### 2. 显式化门槛

根据尺度决定每个维度需要的文档化程度：

| 尺度 | Why 需要 | What 需要 | How 需要 |
|------|---------|----------|---------|
| 战略层 | 决策记录 + 审议 | 产品路线图 | Sprint 规划 |
| 功能层 | 决策记录（轻量） | PRD 文档 | 任务列表 |
| 实现层 | 口头/评论即可 | 接口定义 | 子任务 |
| 代码层 | 不需要 | 不需要 | 直接做 |

### 3. 缺维度检查（推进受阻时执行）

| 症状 | 缺失维度 | 处方 |
|------|---------|------|
| 做着做着不知道为什么做了 | Why 缺失 | 停下来，补决策记录 |
| 知道要做，但不知做到什么程度算完 | What 缺失 | 停下来，补需求定义和验收标准 |
| 目标清楚、标准清楚，但不知怎么下手 | How 缺失 | 停下来，拆任务或寻求技术方案 |
| 三个都有，但互相矛盾 | 对齐缺失 | 回到上一尺度重新审视 |

### 4. 维度回溯（某维度变化时执行）

```
Why 变了 → 检查 What 是否还成立 → 检查 How 是否还有效
What 变了 → 检查 Why 是否还支撑 → 检查 How 是否要调整
How 受阻 → 检查 What 是否定义过度 → 检查 Why 的前提是否还对
```

正交的好处：一个维度变了，另外两个可能完全不用动。先检查，再决定。

### 5. 追溯完整性

每个需求必须能追溯到至少一个决策，每个任务必须能追溯到一个需求或决策。
追溯不到 → 有隐式决策需要显式化。

## 输出格式

```yaml

入口维度: Why|What|How
尺度: 战略|功能|实现|代码
三维状态:
  Why: "{决策状态或'缺失'}"
  What: "{需求状态或'缺失'}"
  How: "{任务状态或'缺失'}"

```
